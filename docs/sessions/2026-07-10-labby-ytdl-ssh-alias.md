# Labby ytdl-rmcp SSH alias session

## User Request

Determine where SSH keys live for `ytdl-rmcp` running inside the Labby Incus container, then configure the Labby-managed MCP server so SSH resolution is owned by the container user's SSH config rather than a host-specific absolute key path.

## Session Overview

The session inspected the live `labby` Incus container, confirmed that `ytdl-rmcp` runs as the `labby` user, found the SSH key material at `/home/labby/.ssh/`, and updated the Labby gateway config so `ytdl-rmcp` uses `YTDLP_REMOTE = "tootie"` with no explicit `YTDLP_SSH_OPTS`. Labby was restarted and the ytdl upstream was verified through a read-only `youtube_stats` call.

## Sequence of Events

1. Inspected Labby CLI help and the live Incus container state for the `labby` container.
2. Confirmed `labby.service` runs as user `labby` and that `ytdl-rmcp` was one of its upstream MCP child processes.
3. Listed the in-container SSH directory and confirmed `/home/labby/.ssh/id_ed25519`, `/home/labby/.ssh/id_ed25519.pub`, `/home/labby/.ssh/config`, and known-host files exist.
4. Read the `ytdl-rmcp` upstream block in `/home/labby/.labby/config.toml` and found the old `YTDLP_SSH_OPTS` pointed at missing path `/home/jmagar/.ssh/id_ed25519`.
5. Verified `ssh tootie true` succeeds as the `labby` user using the SSH host alias.
6. Backed up `/home/labby/.labby/config.toml`, changed the `ytdl-rmcp` upstream to use `YTDLP_REMOTE = "tootie"`, removed `YTDLP_SSH_OPTS`, and restarted Labby.
7. Verified Labby health, gateway runtime state, ytdl tool discovery, and a successful read-only `youtube_stats` call.

## Key Findings

- `labby.service` runs as the `labby` user with main process `/usr/local/bin/labby serve`.
- The Labby Incus SSH bootstrap creates SSH material under `/home/labby/.ssh/`, with the private key at `/home/labby/.ssh/id_ed25519`.
- `/home/labby/.ssh/config` contains a `Host tootie` entry with `User root`, `Port 29229`, and `IdentityFile ~/.ssh/id_ed25519`.
- The old upstream block at `/home/labby/.labby/config.toml:607` used `YTDLP_REMOTE = "root@tootie"` and `/home/labby/.labby/config.toml:609` used a missing host path in `YTDLP_SSH_OPTS`.
- OpenSSH expands `~` for `IdentityFile`, but the cleaner configuration is to let the `Host tootie` alias own user, port, and key resolution.
- Lumen semantic search was attempted first for repo discovery, but its embedding backend reported `all embedding servers are unhealthy`, so live shell inspection was used.

## Technical Decisions

- Prefer `YTDLP_REMOTE = "tootie"` over `root@tootie` plus `YTDLP_SSH_OPTS`, because `/home/labby/.ssh/config` already owns user, port, key, and host-key policy for the container runtime.
- Remove the explicit key path instead of replacing it with `/home/labby/.ssh/id_ed25519`, because the SSH config is generated by `labby setup incus-ssh` and should be the single place for host alias details.
- Restart Labby after editing config so the gateway reloads and reconnects the upstreams with the changed environment.
- Use a read-only ytdl tool call for verification, avoiding any media download or remote write.

## Risks and Rollback

The live Labby config was changed outside this git repo. Rollback is to restore the backup:

```bash
incus exec labby -- cp -a /home/labby/.labby/config.toml.bak.20260710T061911Z-ytdl-ssh-alias /home/labby/.labby/config.toml
incus exec labby -- systemctl restart labby
```

The main risk is if the `tootie` SSH alias changes or is removed from `/home/labby/.ssh/config`; in that case ytdl transfers would fail until the alias is restored or explicit SSH options are reintroduced.
